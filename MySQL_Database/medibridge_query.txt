Why Patients Don't Need UUIDs (Unlike Doctors)
Earlier, we looked at why doctors use strings to hide their internal sequential order from profile page links. The security risks for patient data are handled differently:

No Public Profiles: Unlike a doctor whose information is visible to anyone searching the platform, a patient's dashboard is strictly private. There is no public URL like medibridge.com/patients/101 for an attacker to guess or scrape.

Role-Based Access Control (RBAC): Access to patient records is locked down by security configurations at the backend API layer. Even if someone guesses that patient_id = 45 exists, the backend validates the user's secure active session first. If the logged-in user isn't that exact patient or an authorized doctor, the server blocks the request with a 403 Forbidden response.

This is the action rule that tells MySQL what to do when a parent record is deleted.

If a doctor decides to permanently delete their account from MediBridge, and their unique identifier is removed from the main doctor table, MySQL will automatically go through every other table containing this constraint and instantly delete all rows linked to that specific doctor.
